# Kairo (카이로) — PRD (Product Requirements Document)

> **Knowledge Base가 하나의 파일로, 에이전트는 그걸 읽고 자란다.**
> Karpathy의 LLM Wiki 철학 + DeepSeek V4 Flash 1M Context의 만남

---
## 1. 프로젝트 개요

### 1.1 비전 (Vision)
_"하나의 파일로 시작해, 사용자와 함께 성장하는 개인 에이전트"_

### 1.2 철학
기존 RAG 시스템은 복잡합니다. 임베딩 모델, 벡터 DB, 청킹 전략...  
**Kairo는 다릅니다.** 모든 지식을 하나의 마크다운 파일(`KB.md`)에 저장하고,  
LLM의 긴 컨텍스트(1M tokens)에 통째로 넣어 처리합니다.

> _"RAG를 구축할 시간에, KB.md를 채워라."_

### 1.3 핵심 영감
| 출처 | 영감 |
|------|------|
| **Karpathy LLM Wiki** | 하나의 파일 = 전체 지식, 단순함이 곧 장점 |
| **DeepSeek V4 Flash** | 1M context, $0.x/M tokens → 모든 지식을 통째로 |
| **Hermes Agent** | 자율주행/스킬/크론 개념, Self-improvement 철학 |

## 2. 아키텍처

```
                      ┌──────────────────────┐
                      │      KB.md           │
                      │  (하나의 마크다운)     │
                      │  1M tokens 이하 유지  │
                      └──────────┬───────────┘
                                 │
                      읽기/쓰기   │
                                 ▼
┌──────┐    입력    ┌──────────────────────────┐
│ 사용자 │────────▶│   Kairo Agent (LLM)      │
└──────┘          │   DeepSeek V4 Flash      │
                  │   1M context window       │
                  └──────────┬───────────────┘
                             │
                ┌────────────┼────────────┐
                ▼            ▼            ▼
         ┌──────────┐ ┌──────────┐ ┌──────────┐
         │  답변     │ │ KB.md   │ │ 스킬     │
         │  생성     │ │ 업데이트 │ │ 실행     │
         └──────────┘ └──────────┘ └──────────┘
```

### 2.1 작동 방식

1. **LLM이 모든 걸 처리한다**
   - KB.md 전체를 컨텍스트에 포함
   - 사용자 질문 + KB.md + 시스템 프롬프트 → DeepSeek V4 Flash
   - LLM이 KB.md를 읽고, 판단하고, 기록하고, 실행한다

2. **KB.md는 지식의 전부**
   - 사용자 정보, 프로젝트 지식, 스킬 정의, 크론 설정
   - 대화를 통해 LLM이 "이건 기록해둬야겠다" → 자동 추가
   - "이 지식은 저 지식과 연결된다" → 자동 관계 발견

3. **RAG가 필요없다**
   - 임베딩 모델 ❌ | 벡터 DB ❌ | 청킹 전략 ❌
   - **1M context = 모든 지식을 한 번에**

## 3. KB.md (Knowledge Base) 구조

```markdown
# Kairo Knowledge Base

## 👤 User Profile
- name: 곰탕
- major: Computer Science
- preferences: ...

## 📚 Project: 오픈소스소프트웨어
- team: ...
- deadline: 2026-05-29

## 🔧 Skills
### skill: web-research
- trigger: "검색", "찾아봐", "search"
- action: web_search(query)
- description: 웹 검색 결과 요약

### skill: planner
- trigger: "계획", "일정", "schedule"
- action: generate_plan(tasks)
- description: 작업 계획 수립

## 🧩 Knowledge Graph (자동 발견)
### Edge: 관심사 → 프로젝트
- source: User Profile.관심사
- target: Project.주제
- type: 관련
- discovered: 2026-05-22

## 📊 Growth Log
### Interaction 42 — Agent Level 2 달성!
- date: 2026-05-22
- milestone: 30회 상호작용 완료
- new ability: 동적 크론 추천
```

### KB.md 크기 관리 전략
- **트리거/임계값**: 매 대화 종료 후 80%(800K tokens) 초과 시 LLM이 오래된 지식 자동 요약/압축
- **중요도/안전장치**: Tier1(사용자)>Tier2(스킬)>Tier3(프로젝트)>Tier4(성장), `threading.Lock`으로 직렬화, `.kb.md.bak` 자동 백업
- **초기화/분배**: KB.md 없으면 기본 템플릿 자동 생성, KB.md<80% / 대화<15% / 시스템<5%

## 4. 자율주행 레벨 시스템
### 4.1 자율주행 레벨
| 레벨 | 해금 조건 | 능력 |
|:---:|------|------|
| Lv.0 | 초기 상태 | 기본 에이전트 3종 + 수동 크론 등록 |
| Lv.1 | 10회 이상 상호작용 | 동적 크론 추천 활성화 |
| Lv.2 | 30회 이상 + 크론 3회 이상 수락 | 사용자 의도 예측 제안 |
| Lv.3 | 50회 이상 + 연속 7일 사용 | 선제적 액션 제안 |
| Lv.4 | 향후 확장 (MVP 범위 밖) | 완전 자율 |
### 4.2 정적/동적 크론
| 구분 | 설명 | 예시 |
|------|------|------|
| **정적 크론** | 사용자가 명시적으로 설정한 반복 작업 | "매일 오전 9시 뉴스 요약", "주간 보고서 생성" |
| **동적 크론** | 에이전트가 스스로 판단하여 생성한 작업 | "사용자가 매일 날씨를 묻네? → 아침 날씨 알림 자동 생성" |
### 4.3 빅 스킬 / 마이크로 스킬

| 구분 | 설명 | 예시 |
|------|------|------|
| **빅 스킬** | 복잡한 워크플로우, 여러 마이크로스킬 조합 | "여행 플래너" = 검색 + 일정 + 날씨 |
| **마이크로 스킬** | 단일 동작 단위, 동적 크론의 실행 단위 | 웹검색, PDF 요약, 번역, 알림 |
### 4.4 지식 자율 결합
```
[KB.md에 새 지식 추가]
        ↓
[LLM이 기존 지식과의 관계 감지 + confidence score]
        ↓
├── high (0.9+): 자동 결합 → KB.md Knowledge Graph에 Edge 추가
├── medium (0.7~0.9): 사용자에게 "연결됨" 제안 → 수락 시 추가
└── low (0.7 미만): 무시 (별도 저장 없음)
```

## 5. 기능 요구사항

### 5.1 MVP (1주일 버전)

| 우선순위 | 기능 | 설명 |
|:---:|------|------|
| 🅿️ | **Streamlit 채팅 UI** | 사용자와 에이전트의 대화 인터페이스 |
| 🅿️ | **KB.md 관리** | 파일 읽기/쓰기/업데이트 자동화 |
| 🅿️ | **DeepSeek V4 Flash 연동** | Jiminbox API로 LLM 호출 |
| 🅿️ | **기본 에이전트 3종** | 웹리서치, 플래너, 코딩 도우미 |
| 🅿️ | **자율주행 레벨 시스템** | 상호작용 기반 레벨업 |
| 🅿️ | **README + 발표 PPT** | 오픈소스 활용 과정 명시 |
| ✨ | **KB.md 그래프 시각화** | 지식 연결 관계를 시각적으로 표시 |
| ✨ | **동적 크론 잡** | 사용자 패턴 기반 자동 작업 생성 |
| ✨ | **마이크로스킬 편집기** | UI에서 스킬 추가/수정 |

### 5.2 시스템 설계 고려사항
#### 에러 핸들링 & 복구
| 상황 | 대응 | 복구 |
|------|------|------|
| DeepSeek API 타임아웃 (30초) | 지연 메시지 | 자동 재시도 1회 |
| DeepSeek API 중단 | "서비스 일시 이용 불가" + 캐시 응답 | 30초 간격 재연결 |
| KB.md 파싱 실패 | 오류 로그 | `.kb.md.bak`에서 복구 |
| 크론 작업 실패 | 로그 기록 + 사용자 알림 | 최대 3회 재시도 |
| 동시 쓰기 충돌 | `threading.Lock` 대기 | 잠금 해제 후 실행 |
#### 세션 관리
- **인메모리/영속/복원**: `st.session_state` → KB.md 저장 → 페이지 복원
### 5.3 Streamlit 페이지 구조

    Kairo/
    ├── app.py              # 메인 채팅 인터페이스
    ├── pages/
    │   ├── 1_📚_KB.py      # KB.md 뷰어 / 편집기
    │   ├── 2_🤖_Agent.py   # 에이전트 관리 (스킬 정의/수정)
    │   └── 3_⏰_Cron.py    # 크론 잡 관리 (등록/조회/삭제)
    ├── kb.md               # Knowledge Base
    └── .env                # API 키
### 5.4 기술 스택

| 계층 | 기술 | 사유 |
|------|------|------|
| **Frontend** | Streamlit | 빠른 프로토타이핑, Python 기반 |
| **LLM** | DeepSeek V4 Flash (via Jiminbox API) | 1M context, 초저비용 |
| **지식 저장** | KB.md (단일 마크다운 파일) | RAG 불필요, 단순함 |
| **크론** | APScheduler (Python) | 동적 작업 스케줄링 |
| **스킬 시스템** | Python 함수 + KB.md 정의 | 마이크로스킬 = 함수, 빅스킬 = 조합 |
| **배포** | Streamlit Community Cloud | 무료, 간편 배포 |

## 9. 비교: RAG 방식 VS Kairo 방식

| 기준 | RAG (기존) | Kairo (선택) |
|------|:--------:|:----------:|
| 구축 시간 | 2~3일 | **몇 시간** |
| 임베딩 모델 | 필요 | **불필요** |
| 벡터 DB | 필요 (ChromaDB 등) | **불필요** |
| 청킹 전략 | 필요 | **불필요** |
| 유지보수 | 복잡 | **KB.md 하나만 관리** |
| LLM 비용 | 낮음 (임베딩 + 적은 토큰) | **약간 높음 (전체 KB.md)** |
| DeepSeek V4 Flash | 0.1M tokens = $0.0x | 1M tokens = $0.0x |
| 설명 난이도 | 어려움 | **쉬움 ("그냥 파일 통째로...")** |
| 발표 임팩트 | "열심히 했네" | **"와, 이게 되네?"** |
